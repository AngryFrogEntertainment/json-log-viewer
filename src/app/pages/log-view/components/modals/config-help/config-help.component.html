Configuration Help

This configuration defines how your log entries are interpreted. Basically the parser just parses every line in your log file into an js object. Now you need to tell it where it can find the rigt properties to display what you want to see.

  • Profile: You can create multiple profiles for different formats of your logfiles. "+" Adds a new one.
  • Timestamp: The name of the property that is representing the timestamp. It will be displayed in the list.
  • Level: The name of the property that is representing the used log level. It will be displayed in the list.
  • Message: The log message of your object. Yup, also displayed in the list.
Optional properties
  • Meta informations: Sometimes you need more than just a message to find a bug. So you log objects like errors or current states of your model etc. Here you can tell the log viewer the name of that property. It will be displayed in a modal when you click oon the entry in the list. If it is not defined the complete log entry will be displayed.
  • Context: We like to log the class name where the log was written as context of the specific message. If you do too, then this is your chance to tell the log viewer where. It will be displayed in the list and the modal. If it is not defined you will not see anything.
Formatting
  • Date format: Since we all donÂ´t live in the same country and we have different ways to display a date, you can tell the log viewer how you like it. It has nothing to do with the date format inside of your log file. It is just for display purposes. As long as your file contains a parsable date string, it will do. If not, you got a problem.
  • Page size: You can define here how many log entries you want to see on one page. It has a slight impact on performance. The more there is to be rendered at once, got it?

Note: Except for the meta informations all properties are expected to be a string. If not you will get a "Invalid key" replacer.
